# 维护、拆解与复用工作流

## 内容层

1. 先把候选人的履历拆成“事实、编辑解释、艺术演绎”三类；只有事实进入成绩、年份、职务。
2. Projects 固定为少量旗舰故事，每个故事用三张同语义视觉，不用随机 AI 海报填槽。
3. Lab 只承载公开材料可以引出的未封闭问题，并在卡片上标注来源性质。
4. Social 默认只给四个通用占位；收到真实客户链接后只改 `sources/data/social.js`。

## 视觉层

1. 先冻结 6–8 个颜色令牌与两个字体层级，再生成 UI、信息图、贴图和 3D 材质。
2. 所有压缩纹理宽高必须为 4 的倍数；生成器必须在编码前主动拒绝不合规尺寸。
3. 3D 首先满足固定相机的可读轮廓，再考虑 Blender 正视图；至少验收桌面出生点、窄屏出生点与传送落点。
4. 车辆换皮必须保留 `chassis`、`bodyPainted`、`wheelContainer`、`wheelCylinder` 等节点契约，物理由运行时拥有，模型不自带新物理。

## 工程层

1. 用 `Game.js` 集中注册资源，用 Area 类只做摆放和交互，不把内容数据写进物理逻辑。
2. Career 这类位置驱动系统应映射新年份到既有道路，不为履历长度盲目扩展世界几何。
3. 每个候选人独立分支、起点标签、正式标签和独立发布子路径；根站保持可回退。
4. 每次构建依次通过数据、Social、Landing、Vehicle、3D、发布路径六层校验，再进入浏览器 QA。

## 可复用验收顺序

```text
权利与事实边界
  → 内容清单与视觉母题
  → 确定性二维资产
  → 固定镜头 3D 资产
  → 运行时最小接入
  → 自动校验与生产构建
  → Chrome/WebGPU 桌面 QA
  → 地图逐区跳转 QA
  → 390×844 窄屏 QA
  → 独立路径发布与旧版回退验证
```

## 本轮经验

- 最有价值的个性化不是“放更多东西”，而是让同一个人物命题贯穿入口、车、道路、看板、Lab 和时间机器。
- 确定性 SVG/KTX 生成比随机生图更适合事实看板；生成式图像只用于独立的传播封面。
- 浏览器视觉 QA 必须使用真实 GPU 路径；文件存在和构建成功无法发现压缩纹理块尺寸错误。
- 固定相机项目中，模型朝向、世界坐标和窄屏裁切应写进自动验证，而不是只靠人工记忆。
