# 真实人物动态网站定制工作流 v0.1

> 适用项目：Wen13 3D 动态个人网站及其客户化版本
> 文档类型：甲乙方协作、Agent 执行与验收标准
> 版本：0.1
> 日期：2026-07-22
> 状态：等待首个真实人物样板验证
> 上位依据：`docs/个人定制化内容探索报告.md`

## 1. 工作流目标

本工作流用于把一位真实人物的公开经历、项目证据、思考方式和视觉特征，转译为现有 3D 动态网站中的一套可信、统一、可探索的个人版本。

交付目标不是“填满所有区域”，而是让陌生访问者在一次体验后能够：

1. 说出人物是谁、现在在做什么；
2. 记住至少三个鲜明特征；
3. 理解至少一个可信能力和一个关键人生转折；
4. 区分事实、编辑性概括和艺术化演绎；
5. 在固定镜头、手机和完整昼夜周期中顺利看清核心内容。

本流程也要让下一名 Agent 在不依赖口头记忆的情况下，能够从客户资料开始，独立完成研究、策展、设计说明、资产合同、实施交接和验收。

## 2. 本轮适用范围与红线

### 2.1 允许范围：仅 L0–L2

| 层级 | 本轮允许内容 | 例子 |
|---|---|---|
| L0 内容数据 | 文案、条目、排序、公开链接占位、图片引用、项目和 Lab 数据 | 重写首页介绍、履历节点、3 个项目、4–6 个 Lab 条目 |
| L1 DOM / UI / 贴图 | 菜单、弹窗、卡片版式、字体、颜色、图标、履历贴图、现有模型材质 | 项目看板视觉主题、昼夜两套 UI 对比度、车辆与建筑换色 |
| L2 3D 视觉资产 | 静态模型、低成本循环动画、材质、个人雕像、通用社交图腾、现有物件换皮 | 入口人物雕像、车皮、代表物、现有雕塑替换 |

本轮默认交付必须至少包含一座人物雕像，并可选择少量与人物命题强相关的 L2 代表物。L2 资产只能使用现有区域、现有挂点和现有交互方式，不得反向要求增加新场景或新玩法。

### 2.2 明确冻结：L3–L4

以下需求即使视觉上看似简单，也必须登记为范围变更，本轮不默认实施：

- 新建或重做 Area、地图分区、道路、赛道或室内场景；
- 把保龄球改成羽毛球等新的操作、物理和胜负循环；
- 修改车辆物理、镜头控制、输入逻辑、成就状态机；
- 宠物寻路、跟随、对话、任务或复杂动画状态；
- 实时社媒数据、登录、留言审核、内容后台、多人服务；
- 真实住宅全屋建模、摄影测量大场景或高成本数字孪生；
- 任何要求新后端、长期运维或改变全站引擎契约的功能。

一旦需求触及以上内容，执行 Agent 必须停止该项生产，在变更单中标记为 L3 或 L4，并说明影响、依赖和独立验收条件。不得通过“先顺手做一下”的方式让本轮范围失控。

### 2.3 社交媒体的本轮处理

本轮不把客户的自媒体账号、官方网站或实时平台内容作为个性化核心。Social 区保留小红书、B站、公众号、邮箱等少量通用模型：

- 模型可统一换色或适配整体材质；
- 链接可保留占位或在客户授权后填写；
- 不为每个平台重新设计专属场景；
- 不拉取粉丝数、最新内容或实时数据；
- 不以账号存在与否作为候选人的入选条件。

项目证据中的真实照片、截图、手稿和作品仍然重要。这里冻结的是“社媒/官网深度集成”，不是取消事实证据。

### 2.4 非目标

- 不复制名人网站、媒体文章或受版权保护图片作为交付素材；
- 不用 AI 概念图冒充真实项目证据；
- 不伪造客户、合作、获奖、数据或第三方评价；
- 不默认发布未经本人授权的名人肖像版本；
- 不为了填充版面虚构 8 个项目和 13 个 Lab；
- 不在此阶段建设通用 CMS 或拖拽式建站系统。

## 3. 角色与责任

同一名 Agent 可以承担多个角色，但每个责任必须有明确所有者。

| 角色 | 核心责任 | 不可自行越权的事项 |
|---|---|---|
| 客户 / 肖像权主体 | 提供资料、纠正事实、划定公开边界、确认人物命题与最终肖像 | 未提供的第三方授权不能被默认存在 |
| 项目负责人 | 冻结范围、分配任务、组织评审、裁决冲突、接受或退回成果 | 不得绕过事实与授权闸门上线 |
| 事实研究员 | 建立来源、时间线、事实账本和待确认问题 | 不把推断写成事实 |
| 内容策展人 | 提炼人物命题、编排履历、项目和 Lab | 不用文案修饰掩盖证据缺口 |
| 视觉设计负责人 | 建立颜色、材质、字体、UI、贴图和图像规范 | 不在视觉阶段改变已冻结事实 |
| 3D 资产负责人 | 按雕像合同建模、优化、导出并自检 | 不自行改变肖像、姿态或授权范围 |
| 集成负责人 | 将 L0–L2 内容和资产接入现有位置，保留回退点 | 不新增 Area、状态机或后端 |
| QA / 验收人 | 做事实、固定镜头、手机、昼夜、性能和回归检查 | 不以“本机看起来没问题”代替证据 |
| 资产权利负责人 | 维护授权、许可证、署名和发布状态清单 | 证据不全的资产不得标记为可发布 |

若采用多 Agent：研究 Agent 只提交事实包；策展/设计 Agent 只能使用已标记的事实和授权；集成 Agent 只能使用已通过设计评审的清单；最终接受必须由项目负责人执行。

### 3.1 子智能体监督闭环

子智能体“完成”只表示已提交候选产出，不表示闸门通过。主 Agent 必须执行以下闭环：

1. **分配**：用书面任务说明输入、唯一输出目录、禁止改动、事实边界和验收清单；不同 Agent 不得同时编辑同一交付文件。
2. **中期检查**：在候选筛选、人物命题、空间映射、雕像灰模等高返工节点检查一次可见成果；发现越过 L2、来源薄弱或模板化视觉时立即纠偏。
3. **独立复核**：主 Agent 对事实引用、区域映射、资产权利和 QA 证据抽样回读，不能只转述子智能体的自评。
4. **退回**：未满足标准时，主 Agent 给出“问题—证据—修改目标—不应改变的部分—复验方式”，再用后续任务触发同一 Agent 修订。
5. **再验收**：修订稿必须重新通过原闸门；不能因为“已经改过一次”而降低标准。
6. **合并**：主 Agent 重读目标文件的最新版本，处理并行产出的冲突后再写入共享索引、项目日志和最终交付。
7. **结项**：只有主 Agent 在交付清单中逐项签收后，子任务结果才能计入正式成果。

监督记录至少包含：任务 ID、负责人、输入版本、输出路径、中期反馈、退回原因、修订版本、复验结论和最终签收人。

## 5. 客户资料最小包

资料最小包不是“有多少就全放进去”，而是达到人物命题、3 个旗舰故事和一座雕像可成立的最低输入。

### 5.1 必需资料

| 类别 | 最低输入 | 用途 | 通过条件 |
|---|---|---|---|
| 基本身份 | 公开姓名/称呼、当前身份、所在地到城市级、希望被如何介绍 | Landing、主页、关于 | 客户确认公开表述 |
| 可核验时间线 | 5–7 个关键节点，含时间、地点、角色和变化 | Career、Time Machine | 每个节点有来源或本人确认 |
| 旗舰故事 | 3 个项目、作品、创业、研究或生活工程 | Projects | 能说明本人角色、选择、结果和证据 |
| Lab 材料 | 4–6 个正在思考、学习、实验或未来方向 | Lab | 不与已完成项目重复，且客户愿意公开 |
| 人物访谈 | 当前关心的问题、方法、价值观、转折、失败与反思 | 人物命题、文案口吻 | 至少形成一份经确认的访谈摘要 |
| 雕像参考 | 正面、左右侧、三分之四、全身或半身照片；身高/比例、典型服装、配件 | L2 雕像 | 照片来源明确，肖像使用范围明确 |
| 视觉偏好 | 喜欢/不喜欢的颜色、材质、风格、物件和参考案例 | 视觉系统 | 能形成 3–5 个视觉关键词和禁用项 |
| 授权边界 | 可公开、需脱敏、仅内部参考、禁止使用四类清单 | 全部区域 | 客户或权利主体签字/书面确认 |

### 5.2 旗舰故事的最低证据

每个旗舰故事至少要有：

- 时间、角色和本人贡献；
- 一个真实问题或处境；
- 一个关键决定或行动；
- 一个可核验结果或明确变化；
- 至少一项有权使用的真实物证：产品截图、现场照、手稿、公开记录、文档节选或成品；
- 一段本人确认的反思。

若不能满足，故事只能进入“待补证”或 Lab，不能作为旗舰项目发布。

### 5.3 本轮不要求

- 不要求真实小红书、B站、公众号或官网主页；
- 不要求实时账号数据；
- 不要求全屋测量、宠物完整动作资料或新玩法需求文档；
- 不要求把所有职业和生活经历全部公开。

## 6. 事实、推断、演绎与授权分层

### 6.1 内容事实层

所有人物内容使用以下标签，不允许只写“已查到”：

| 标签 | 定义 | 可以怎样使用 | 发布要求 |
|---|---|---|---|
| F1 一手公开事实 | 本人、所属组织、原始作品、监管/学校等权威记录直接发布 | 可写入主叙事 | 保留链接、日期和摘录摘要 |
| F2 可靠二手事实 | 可靠媒体、出版物、经核实采访转述 | 可辅助主叙事 | 至少记录来源和与一手材料的交叉情况 |
| F3 本人确认事实 | 客户在访谈或书面中确认、但未公开 | 仅按授权范围使用 | 保存确认方式和公开级别 |
| I1 编辑性推断 | 根据多个事实得出的叙事判断，如“持续重新定义工作方式” | 必须写成解释或提案，不能伪装成客观事实 | 客户确认后才进入核心文案 |
| A1 艺术化演绎 | 雕像姿态、空间隐喻、色彩人格、未来来信等设计表达 | 可用于视觉与体验 | 显著避免让人误认为真实事件 |
| U 未确认 | 单一传闻、时间冲突、无法核实或仍待客户回答 | 只能进入问题清单 | 不得发布 |

### 6.2 发布授权层

事实真实不等于拥有发布权。每项内容还必须独立标记：

| 标签 | 含义 | 处理 |
|---|---|---|
| P0 公开可用 | 已公开且使用方式符合来源/许可 | 可进入发布候选 |
| P1 客户授权 | 本人拥有权利并明确授权本项目使用 | 按授权介质、范围和期限使用 |
| P2 第三方待授权 | 含合作方、家人、同事、客户、摄影师、商标或保密信息 | 获书面许可或脱敏后再使用 |
| P3 仅内部参考 | 可用于理解、情绪板或建模参考，但不能对外展示 | 不进入公开构建和交付媒体包 |
| P4 禁止使用 | 客户明确禁止或法律/伦理风险不可接受 | 从生产清单排除 |

发布条目的最小组合为 `F1/F2/F3 + P0/P1`。`I1` 和 `A1` 必须同时有客户确认；`U`、`P2`、`P3`、`P4` 均不能进入公开版本。

### 6.3 事实账本字段

```text
事实 ID：FACT-001
内容摘要：
时间范围：
涉及人物/组织：
内容层级：F1 / F2 / F3 / I1 / A1 / U
来源 URL 或受控证据位置：
访问/确认日期：
本人角色与贡献：
发布授权：P0 / P1 / P2 / P3 / P4
可公开表述：
禁止表述：
消费区域：Landing / Career / Projects / Lab / 其他
复核人：
状态：候选 / 已核验 / 待客户确认 / 禁用
```

### 6.4 名人研究的额外边界

名人公开资料适合验证策展流程，但“资料公开”不等于可使用其肖像、照片、商标、作品或背书。未经本人或权利方授权：

- 只能产出内部研究、低保真映射和原创风格探索；
- 不得公开发布可被误解为本人官网、合作或认可的成品；
- 雕像只可作为不发布的概念验证，或改为明确非商业、非公开的抽象占位；
- 最终商业模板应转向已授权真实客户或全面模拟人物。

## 7. 候选人或客户筛选

### 7.1 筛选评分表

每项按 0–5 分评分，并写证据，不只写主观印象。

| 维度 | 判断问题 | 建议权重 |
|---|---|---:|
| 事实完整度 | 是否有可交叉核验的时间线和本人角色？ | 20% |
| 旗舰故事密度 | 是否有至少 3 个处境—行动—结果完整的故事？ | 20% |
| 人物命题潜力 | 经历之间是否存在可概括但不扁平的主线？ | 15% |
| Lab 潜力 | 是否有持续问题、阅读、实验或未来方向？ | 10% |
| 视觉象征 | 是否有独特服装、姿态、物件、色彩或空间隐喻？ | 10% |
| L0–L2 可实现性 | 不建新场景、不改玩法能否表达其魅力？ | 10% |
| 权利可控性 | 肖像、项目材料、第三方和商标风险是否可管理？ | 10% |
| 客户协作度 | 能否持续确认事实、选择与授权？ | 5% |

建议入选条件：总分达到项目方设定阈值，且“事实完整度”“旗舰故事密度”“权利可控性”“L0–L2 可实现性”均不得低于 3 分。具体阈值在样板后冻结，v0.1 不用未经验证的总分替代专业判断。

### 7.2 一票否决

- 核心经历主要来自无法核实的网络传闻；
- 项目成果无法区分个人与团队贡献；
- 需要虚构或夸大才能形成魅力；
- 只有社交账号，没有可编辑的真实故事；
- 必须新增大场景或玩法才能成立；
- 肖像或第三方材料的发布权无法取得；
- 候选人处于可能被消费的重大创伤、争议或安全风险中；
- 客户拒绝事实复核、隐私分层或关键授权。

### 7.3 候选比较输出

候选研究必须提交：

1. 评分矩阵和证据；
2. 每人一句暂定人物命题；
3. 3 个可能的旗舰故事；
4. 4–6 个可能的 Lab 主题；
5. 一座雕像的视觉方向；
6. L0–L2 内的空间映射草案；
7. 权利、争议和事实缺口；
8. 推荐一人及不选其他人的理由。

候选确定不是生产授权。进入下一阶段前仍需单独通过事实与发布边界闸门。

## 8. 人物核心命题

### 8.1 命题不是口号

人物命题是对一组已核实人生选择的编辑性解释。推荐句式：

> 他/她是一个【身份与行动方式】的人，通过【反复出现的方法或选择】，持续解决【长期关心的问题】，即使面对【真实张力或代价】。

命题必须由至少三个不同人生阶段的事实支撑，且能同时指导文字、视觉、空间和雕像。只写“连续创业者”“创新者”“终身学习者”不够，因为任何人都可以使用。

### 8.2 人物命题文件

`character-thesis.md` 至少包含：

- 一句话命题；
- 3–5 个支撑事实 ID；
- 一个不能被忽略的矛盾或张力；
- 三个希望访问者记住的特征；
- 一个可信能力；
- 一个关键故事；
- 文案口吻的 3 个关键词和 3 个禁用词；
- 视觉关键词和反关键词；
- 客户原话与编辑版之间的差异；
- 客户确认状态。

### 8.3 命题闸门

只有同时满足以下条件才通过：

- 客户认为“像我”，但不是空泛赞美；
- 三个旗舰故事都能服务命题，却不会被强行解释成同一种故事；
- Landing、Career、Projects、Lab 各自承担不同问题；
- 视觉设计能从命题中推导，而不是先选一个流行风格；
- 不依赖社交媒体账号或网站链接才能理解人物。

### 9.3 去重规则

- Career 讲“变化”，Projects 讲“贡献”，Lab 讲“未完成”；
- 同一项目可在 Career 出现一次转折，在 Projects 展开证据，但文案不可重复；
- 荣誉只能解释为项目结果之一，不应成为整个人物命题；
- Time Machine 使用版本差异，不是另一份 Career；
- Behind the Scene 提供完整说明，不承担主体验的首个惊喜；
- Social 只承担联系，不用平台 Logo 填补人物信息不足。

### 9.4 取舍原则

没有强材料的区域可以保持通用、弱化或从直达地图中降级，不能用随机生成内容补齐。首个样板优先保证 Landing、Career、Projects、Lab 与雕像形成闭环。

## 10. 项目看板编排

### 10.1 旗舰项目选择

每个候选项目按以下问题筛选：

- 能否看出本人而非团队整体的贡献？
- 是否存在真实问题、决定和变化？
- 是否有可合法使用的物证？
- 是否补充另两个项目没有展示的能力或性格？
- 是否能在现有多屏、三图、属性与链接结构中表达？

默认选择 3 个强项目，不为占满当前 8 个槽位而生产弱内容。剩余项目可隐藏、合并为时间线节点，或留作后续版本。

### 10.2 单项目编辑规格

每个项目写成六幕：

1. **封面**：一句具体冲突或结果，不是职场套话；
2. **处境**：当时为什么必须做；
3. **角色**：本人负责什么、没有负责什么；
4. **选择**：关键决策、失败、权衡或代价；
5. **证据**：实际产物、过程物证、结果和来源；
6. **余波**：这件事如何改变后续判断或人生方向。

现有三图槽位建议分别承担：

- 图 1：真实结果或成品；
- 图 2：过程、手稿、现场或关键迭代；
- 图 3：影响、反馈、前后对照或反思。

图像不足时可使用原创信息图、排版化文字、经过标注的重建示意图；不得用无依据的 AI 场景图伪装为现场。

### 10.3 三张项目图的强制分类

每张项目图在生产清单和 UI 图注中必须属于以下一类：

| 标记 | 类型 | 允许内容 | 禁止事项 |
|---|---|---|---|
| `P-EVIDENCE` | 事实证据图 | 真实成品、界面截图、现场照片、手稿、公开记录、真实前后对照 | 不得用生成图重画后继续称为真实截图或现场 |
| `P-EXPLAIN` | 原创解释图 | 根据已核验事实制作的流程图、结构图、数据图、重建界面和注释排版 | 必须标明“根据真实资料整理/重建”，不得加入无来源数字 |
| `P-MOOD` | 氛围图 | 原创插图、抽象隐喻、装饰封面、情绪背景 | 不得冒充项目现场、产品界面、客户成果或第三方证据 |

默认三图组合优先使用 `P-EVIDENCE + P-EVIDENCE/P-EXPLAIN + P-EXPLAIN`。只有项目已具备独立事实证据时才允许加入 `P-MOOD`；氛围图不能是某个旗舰项目唯一或主要“证据”。

所有看板交付纹理统一输出为 **KTX2、960 × 540、16:9**，并保留可编辑母版。压缩色彩空间、透明通道和质量参数应在首个样板实测后冻结；同一项目三图必须使用一致设置。源图片进入 KTX2 前仍要保留来源、裁切范围和哈希，不能让转换后的文件失去证据链。

### 10.4 看板视觉个性化

每个项目可以有独立强调色、代表图标和证据排版，但必须共享：

- 字体和字号层级；
- 标题、角色、时间、证据、反思的固定信息位置；
- 翻页、关闭、链接和返回操作；
- 昼夜对比度规则；
- 图片裁切比例与来源标记；
- “真实证据 / 演绎示意”的视觉标签。

项目 UI 的个性来自内容类型，不来自随机更换风格。科技、教育、创作等项目可以改变内部隐喻，但不能破坏整体世界的低多边形与玩具感。

### 10.5 项目闸门

单项目有以下任一情况即退回：

- 看不出本人贡献；
- 结果只有自我评价，没有证据或清晰变化；
- 图片比故事更“宏大”，容易误导；
- 氛围图被放在事实证据位，或图注让访问者误认为真实现场；
- 三张图表达同一件事；
- 与另一个项目展示完全相同的能力；
- 文案需要滚动很久才能理解；
- 手机或夜间无法读清；
- 素材授权状态不是 P0/P1。

## 11. Lab 编排

### 11.1 Lab 与 Projects 的边界

Projects 证明“做成过什么”；Lab 呈现“正在为什么着迷”。Lab 可以不成功、不完整，但必须真实且具体。

建议从以下栏目中选 4–6 项，而不是填满 13 个：

- `Now`：正在进行的实验、练习或观察；
- `Questions`：反复追问但没有标准答案的问题；
- `Inputs`：真正改变判断的一本书、一段课程或一位前辈；
- `Seeds`：草图、笔记、原型和未完成想法；
- `Next`：已承诺的下一步；
- `Contradictions`：本人仍未解决的矛盾；
- `Future Note`：明确标为未来想象的短文本。

### 11.2 单条 Lab 规格

```text
标题：不超过一行
状态：Now / Question / Input / Seed / Next / Contradiction / Future Note
真实起点：对应事实 ID 或客户原话
为什么持续在意：1–2 句
当前进度：具体到一个动作或产物
下一次可观察变化：
主图：笔记/书页/原型/示意，注明真实或演绎
缩略图：与主图同一视觉系统
公开级别：
与项目区的区别：
```

### 11.3 Lab 闸门

- 不能只是“我正在学习 AI”“我爱读书”等空泛兴趣；
- 不能把已完成项目换标题后重复展示；
- 不能把尚未发生的目标写成既有成就；
- 不能公开客户、公司或未发布产品的保密内容；
- 视觉应允许粗糙、过程感和未完成感，但排版仍需统一；
- 每条至少能回答“为什么是这个人在研究它”。

## 12. 视觉系统

### 12.1 视觉系统由人物命题推导

视觉设计开始前先冻结：

- 3 个性格关键词；
- 1 个核心张力，如“精密 / 温度”“克制 / 冒险”；
- 1 件代表物；
- 1 种主要材质和 1 种辅助材质；
- 1 组昼夜都成立的主辅色；
- 3 个禁用方向，防止回到泛化 AI 科技风。

### 12.2 必交视觉令牌

`visual-system.md` 至少规定：

- 可直接取样的色票表：令牌名、sRGB HEX、用途、白天/夜间版本、可搭配材质；
- 主色、辅助色、强调色、危险/成功状态色；
- 白天与夜间背景、文字、描边和发光策略；
- 字体、字号、行距和中英文混排；
- 金属、石材、木材、布料等材质的粗糙度与饱和度范围；
- 线条粗细、圆角、阴影、噪点、图标和二维码风格；
- 图片裁切、边框、色彩处理和证据标签；
- 构图规则：固定镜头中的主体占比、前景遮挡、文字安全区、项目屏幕 16:9 画面和手机裁切安全区；
- 雕像、车辆、社交图腾和 UI 的共享色盘；
- 音频如不定制则明确沿用现有系统；
- 禁止使用的颜色、纹理、字体和效果。

### 12.3 图像生产四分法

| 类型 | 用途 | 标记规则 |
|---|---|---|
| E 真实证据 | 成品、截图、现场、手稿、照片 | 不得改变事实含义；记录来源与裁切 |
| R 信息重建 | 时间图、结构图、数据图、界面重排 | 标为“根据真实资料重建” |
| I 艺术插图 | 项目情绪封面、过渡图、抽象隐喻 | 不得让人误认成真实现场 |
| D 装饰纹理 | 背景、材质、粒子、无事实含义图形 | 可 AI 辅助，但需统一风格与权利记录 |

AI 生图只能主要承担 I 和 D；若用于 R，必须有真实资料输入和人工校验；不能替代 E。

### 12.4 视觉评审闸门

- 只看 Landing 截图也能感到人物差异，而不是换了名字的 Wen13；
- Projects 与 Lab 看起来属于同一个世界，但表达“完成”和“未完成”的语气不同；
- 雕像、UI、车辆换色和环境贴图共享色盘；
- 不依赖大量霓虹、全息面板、机器人或宇宙图制造“AI 感”；
- 白天、夜间、手机截图都通过可读性检查；
- 所有生成资产的用途与事实层级已标记。

## 13. 3D 人物雕像合同

个人雕像是本轮唯一必选 L2 资产。合同的目标是避免“Blender 里好看、进游戏只能看到背面”以及资产过重、肖像不准、权利不清等问题。

### 13.1 设计输入

- 已批准的人物命题和视觉关键词；
- 有权作为参考的正面、左右侧、三分之四、全身/半身照片；
- 典型服装、眼镜、发型、体态和代表物；
- 客户希望强化和不希望夸张的特征；
- 写实、低多边形、雕塑化或其他风格选择；
- 现有世界截图、人物出生点、Wen13 字样及主行车线；
- 明确的肖像和商业使用授权。

### 13.2 设计合同

```text
资产 ID：STATUE-<customer>-01
摆放区域与现有挂点：Landing / 具体节点
模型根节点：默认匹配现有 `landingKnightStatuePhysicalFixed...` 命名契约；若更名必须同步并验证 `LandingArea.setStatue()` 查找规则
视觉/物理节点关系：根节点必须能被现有 `addFromModel()` 读取；不得只交付一个脱离现有物理命名协议的显示网格
人物命题：
风格：
姿态与朝向：
代表物：
材质与色盘：
底座形状与铭文：
允许的艺术夸张：
禁止的夸张：
是否含动画：默认无；若有，仅使用现有可播放循环
碰撞合同：默认沿用一个简化固定碰撞体；碰撞范围不得覆盖出生点、主道路或交互点；复杂高模不得直接充当碰撞网格
肖像授权：
模型/底模/纹理来源：
性能预算：以当前入口雕像和场景基线实测后冻结
导出格式：GLB/glTF，与现有加载链一致
验收镜头：默认固定相机 + 主要行车线 + 手机
验收光照：白天、黄昏、夜间
回退资产：现有雕像或上一个已通过版本
```

### 13.3 建模和导出约束

- 尺寸、坐标轴、原点和缩放与现有 Landing 资产一致；
- 当前入口加载器通过 `/^landingKnightStatuePhysicalFixed/i` 找模型根节点；资产交付必须保留兼容名称，或在同一变更中有明确、可回退的查找规则修改与自动校验；
- 应用变换，删除隐藏高模、无用摄像机、灯光和未使用材质；
- 优先少材质、共享色盘和简单纹理；
- 面数、贴图尺寸、材质数和压缩后体积不能高于冻结预算；
- 使用合法底模时记录许可证、修改权和署名要求；
- 轮廓必须在低分辨率和远距离仍可辨认；
- 面部不追求不必要的写实细节，应与现有低多边形世界统一；
- 若不需要碰撞，不能让复杂网格进入物理系统；
- 使用固定、简化的碰撞体时，站内必须验证小车无法穿入雕像，也不会被碰撞体无故挡在道路或传送落点；
- 所有外部纹理应随 GLB 正确打包或按项目规范引用；
- 不依赖只有建模软件中存在的灯光才好看。

### 13.4 固定镜头朝向合同

当前网站不是自由转镜头体验。雕像正面必须按默认相机的 `+X / +Z` 观看象限和主要行车路线落位，而不是按 Blender 正视图猜测。

至少验证以下视点：

1. 初始出生/加载完成视点；
2. 雕像旁最近可驾驶点；
3. 从主要道路驶入时的首次揭示；
4. 从道路驶离时的侧后轮廓；
5. 手机竖屏与横屏的实际裁切。

必须看清：脸部或头部朝向、代表物、完整轮廓、底座文字。若只能通过开到地图边角或调试自由相机看正面，判定失败。

### 13.5 雕像评审顺序

1. 黑白剪影和姿态草图；
2. 三分之四视图与固定镜头合成；
3. 低模灰模进站截图；
4. 材质与色盘；
5. 白天、黄昏、夜间截图；
6. 手机与性能检查；
7. 肖像主体、项目负责人和权利负责人签收。

若灰模构图未通过，不得进入精修和材质阶段。

- 公司项目截图要检查保密义务、客户名称、个人数据和后台信息；
- 字体、音乐和开源 3D 底模也需检查许可证；
- AI 生成资产需记录模型/服务、输入权利、用途和人工修改；
- 名人的公开肖像不能自动转化为商业雕像授权；
- P2/P3/P4 资产必须被构建前检查阻断，而不是靠人工记忆。

## 15. 结果闸门式执行流程

流程以可验收结果推进，不以天或周排期。

### G0 范围冻结

**输入：** 客户意图、预算边界、当前网站能力。
**输出：** `brief.md`、L0–L2 清单、冻结的 L3–L4 清单、验收语句。
**通过：** 每项需求均有层级；社交只留通用模型；至少一座人物雕像；无新场景/玩法/后端。
**失败处理：** 将越界项写入变更登记，不进入当前生产。

### G1 候选与资料成立

**输入：** 候选来源或客户资料。
**输出：** 筛选评分、资料最小包、事实账本初稿、缺口清单。
**通过：** 5–7 个时间节点、3 个可成故事的项目、4–6 个 Lab 方向和肖像参考基本成立。
**失败处理：** 更换候选、缩小主张或转为内部模拟样板。

### G2 事实与授权成立

**输入：** 来源、访谈、项目材料、照片和授权。
**输出：** 完整事实账本、发布边界、资产权利初表。
**通过：** 核心内容达到 `F1/F2/F3 + P0/P1`；第三方和保密风险已清除或脱敏。
**失败处理：** 降级为 Lab、替换证据、重新表述或删除。

### G3 人物命题成立

**输入：** 已核验事实、访谈、矛盾和视觉偏好。
**输出：** 一句话命题、三项特征、能力、关键故事、口吻和禁用词。
**通过：** 至少三个阶段事实支撑；客户确认；不依赖社媒资料。
**失败处理：** 返回访谈和事实排序，不进入视觉精修。

### G4 人生—空间映射成立

**输入：** 人物命题、当前区域能力。
**输出：** Landing、Career、Projects、Lab 等区域映射及取舍。
**通过：** 每个区域只回答一个主问题；3–4 个项目和 4–6 个 Lab 不重复；第四个项目已证明不可合并且不用于填槽；全部为 L0–L2。
**失败处理：** 合并、隐藏弱内容；越界需求进入变更登记。

### G5 内容原型成立

**输入：** 映射、事实和证据。
**输出：** 首页、履历、项目、Lab、关于页的真实内容原型。
**通过：** 文字、图片用途和证据链通过；即使无精美视觉也能让陌生人理解人物。
**失败处理：** 补证、重写叙事，不用生成图掩盖问题。

### G6 视觉系统与雕像灰模成立

**输入：** 命题、内容原型、现有世界截图。
**输出：** 视觉令牌、看板规范、资产清单、雕像合同和固定镜头灰模。
**通过：** 整体统一；昼夜可读；灰模在默认视角可识别；资产权利路径明确。
**失败处理：** 先修构图和色盘，禁止提前精修。

### G7 L0–L2 生产与集成成立

**输入：** 已签收规范和可发布资产。
**输出：** 内容数据、UI/贴图、雕像及其他批准 L2 资产的可回退集成版本。
**通过：** 构建成功；无 L3/L4 改动；原玩法、地图、控制和外观回退正常。
**失败处理：** 按改动等级回退到最后通过版本，修复后重新进入 G7。

### G8 体验 QA 成立

**输入：** 候选发布构建。
**输出：** 固定镜头、手机、昼夜、内容、权利、性能和回归证据。
**通过：** 第 16 节所有阻断项通过。
**失败处理：** 按问题归属退回内容、视觉、3D 或集成阶段。

### G9 交付与复盘成立

**输入：** 已通过 QA 的构建和所有文档。
**输出：** 交付清单、版本/回退证据、维护说明、经验复盘。
**通过：** 第三方可依据清单复现；客户明确知道以后修改属于 L0–L4 哪一级。
**失败处理：** 缺什么补什么，不以“已经能打开”代替完整交付。

## 16. QA 清单

### 16.1 内容与人物 QA

- [ ] 60–90 秒内可理解人物身份、现在做什么和继续探索的理由；
- [ ] 至少三个鲜明特征可由页面事实支撑；
- [ ] 每个旗舰项目说明本人角色、选择、结果和证据；
- [ ] Lab 均为真实未完成方向，不把愿望写成成就；
- [ ] Career 时间、地点、职务与项目没有冲突；
- [ ] 事实、推断、艺术演绎和 AI 辅助内容可区分；
- [ ] 社交模型为通用入口，不承担未经验证的人物主张；
- [ ] 客户/权利主体完成最终事实复核。

### 16.2 固定镜头 QA

- [ ] 雕像在出生点能看到可识别的正面或三分之四面；
- [ ] 从主要道路驶入时，人物头部、姿态和代表物不会被字牌、树木或建筑遮挡；
- [ ] 靠近后底座文字和人物轮廓仍完整；
- [ ] 从所有可驾驶位置不要求自由相机才能理解关键资产；
- [ ] 新物件的正反面按默认相机 `+X / +Z` 观看象限落位；
- [ ] 模型碰撞不会卡住小车、出生点或传送落点；
- [ ] 地图传送后不会把相机放在模型内部或背面；
- [ ] 默认缩放、窗口缩放和浏览器工具栏变化后构图仍成立。

### 16.3 手机 QA

至少覆盖项目定义的主流小屏和一台真实手机，竖屏、横屏各走一次核心路径：

- [ ] 首屏加载、关闭提示、触控驾驶、菜单、地图和返回可操作；
- [ ] Landing 人物命题、雕像和控制提示不相互遮挡；
- [ ] 项目标题、正文、图片、上一项/下一项和关闭按钮可读可触；
- [ ] Lab 标题、缩略图和导航不会溢出；
- [ ] 长中文断行、英文名、数字和标点正常；
- [ ] 通用社交模型触发区不会重叠；占位链接不误导用户；
- [ ] 页面切后台再回来不会丢失必要状态或持续异常音频；
- [ ] 首次加载和场景切换无明显内存崩溃、黑屏或纹理缺失；
- [ ] 低性能设备仍可使用菜单/地图直达核心信息。

### 16.4 昼夜与天气 QA

至少取白天、黄昏、夜间三个阶段，并在一轮 5 分钟周期中观察过渡：

- [ ] 人物面部/轮廓在三种光照下都可识别；
- [ ] UI、项目图注、履历和底座文字对比度足够；
- [ ] 黑色材质夜间不消失，白色/发光材质夜间不过曝；
- [ ] 雕像阴影、法线和粗糙度无明显异常；
- [ ] 雨、雪、雷电或强环境效果不会长期遮挡关键信息；
- [ ] 颜色变化不改变证据图片原意；
- [ ] 昼夜切换不造成模型闪烁、材质丢失或文字不可读。

### 16.5 工程与回归 QA

- [ ] 生产构建成功，控制台无新增阻断错误；
- [ ] 资产路径、大小写和部署基路径正确；
- [ ] 模型加载失败时有可接受回退，或不会阻断世界进入；
- [ ] 小车移动、碰撞、翻转、重生和鸣笛与改动前一致；
- [ ] 地图 12 个现有传送点、区域触发、保龄球和赛道原逻辑未被破坏；
- [ ] 项目、Lab、Career、菜单和关于页能完成打开、切换、关闭；
- [ ] 通用社交外链为正式目标或明确占位，不产生错误承诺；
- [ ] 资产哈希与权利清单一致；
- [ ] 已保存版本、构建证据和可执行回退方案。

### 16.6 发布阻断项

以下任一项失败即不得标记为可发布：

- 核心事实未确认或存在冲突；
- 肖像、照片、模型、字体、音乐或第三方材料权利不清；
- 雕像默认视角只看到背面或被严重遮挡；
- 手机无法访问核心信息；
- 夜间关键信息不可读；
- 变更实际越过 L2；
- 新资产导致构建失败、主场景无法进入或核心玩法回归失败；
- 未保留上一可用版本与回退证据。

## 17. 评审、返工与变更触发

### 17.1 评审对象和签收人

| 评审 | 必须签收 | 通过证据 |
|---|---|---|
| 事实评审 | 事实研究员 + 客户/权利主体 | 事实账本状态、确认记录 |
| 人物命题评审 | 客户 + 项目负责人 + 内容策展人 | 命题文件与三项支撑事实 |
| 空间映射评审 | 项目负责人 + 内容/设计/集成 | 映射表与 L0–L2 分级 |
| 视觉评审 | 客户 + 视觉负责人 + 项目负责人 | 白天/夜间/手机关键画面 |
| 雕像灰模评审 | 客户 + 3D + 集成 + QA | 固定镜头站内截图 |
| 发布评审 | 项目负责人 + QA + 权利负责人 | QA 清单、权利清单、回退证据 |

### 17.2 必须返工的条件

发生以下情况应退回相应闸门，而不是在后续阶段打补丁：

- 客户指出重要事实、语气或肖像“不像本人”；
- 新证据推翻人物命题或关键项目结论；
- 三个区域重复讲同一故事；
- 视觉风格只能用“高级、科技、未来感”描述，无法联系人物事实；
- 真实证据被生成图取代或与生成图混淆；
- 雕像在固定镜头不可识别；
- 昼夜或手机测试破坏信息理解；
- 新需求需要状态机、物理、Area 或后端；
- 权利清单出现 P2–P4 资产；
- 集成后出现主玩法、地图或控制回归。

### 17.3 不应触发全面返工的情况

- 客户只要求在已冻结色盘内调一个颜色；
- 已确认文案的语法、数字格式或轻微措辞修正；
- 不改变人物识别的局部材质或模型小修；
- 替换同一权利状态、同一用途和同一尺寸的证据图片；
- 修复明确的技术缺陷且不改变已批准设计。

## 20. 常见失败与防范

| 常见失败 | 根因 | 预防机制 | 发现后退回 |
|---|---|---|---|
| 网站像高级模板，不像本人 | 没有人物命题，先选风格后找故事 | G3 命题闸门；三阶段事实支撑 | G3 |
| AI 图很漂亮但不可信 | 用生成图代替项目证据 | E/R/I/D 分类；旗舰项目真实物证最低要求 | G2/G5 |
| 简历很全但没有故事 | 只列职位，没有处境、选择和余波 | 旗舰项目六幕；Career 强调转折 | G5 |
| Lab 是项目废稿或愿望清单 | 没有定义“未完成但真实” | Lab 状态、真实起点和下一变化 | G4/G5 |
| 所有区域重复同一段简介 | 没有给区域分配唯一问题 | 人生—空间映射和去重规则 | G4 |
| 为填满槽位制造弱内容 | 把容量当成目标 | 3 项目、4–6 Lab 优先；允许隐藏 | G4 |
| 名人资料公开就直接上线 | 混淆事实可查与肖像/版权可用 | 双层 F/P 标记；名人额外边界 | G2 |
| 雕像只能看到背面 | 以 Blender 视图代替真实相机 | 灰模进站；`+X / +Z` 固定镜头合同 | G6 |
| 雕像太重或材质失效 | 未在生产前冻结预算和导出规范 | 以现有基线冻结预算；GLB 自检 | G6/G7 |
| 夜间人物和文字消失 | 只在白天做视觉评审 | 日/暮/夜三状态令牌与 QA | G6/G8 |
| 手机上无法关闭或看文字 | 只做桌面截图 | 真实手机竖/横屏核心路径 | G8 |
| 换皮意外破坏碰撞/玩法 | 把 L2 视觉替换当作无需回归 | 保留挂点/碰撞契约；玩法回归清单 | G7/G8 |
| 通用社交模型喧宾夺主 | 平台 Logo 被当人物主内容 | Social 降级为联系入口 | G4/G6 |
| “小改动”演变成新玩法 | 未做变更分级 | 每项进入 change-register；L3/L4 默认延后 | G0/G7 |
| 客户最后才说不像本人 | 只让客户看最终成品 | 命题、内容、灰模、视觉分阶段签收 | 对应最近闸门 |
| 素材散落在聊天和文件名 | 没有单一真源 | 标准目录、事实 ID、资产 ID | G1/G2 |
| 已发布但无法回退 | 只保存最终构建 | 版本、哈希、上一可用版本和回退演练 | G7/G9 |

## 21. 独立 Agent 执行指令

另一名 Agent 接手时，应按以下顺序执行，不得跳过：

1. 读取本工作流、上位探索报告、项目执行日志和当前仓库规则；
2. 确认工作目录、分支、现有改动和当前公开版本，保留所有用户改动；
3. 创建客户目录和执行日志，记录范围为 L0–L2；
4. 建立候选评分与资料最小包，不够则报告缺口；
5. 建立事实账本和发布授权，不使用 U/P2/P3/P4 内容生产；
6. 提交人物命题给负责人评审，通过前不做精修图；
7. 完成人生—空间映射，逐项写出等级和“不做什么”；
8. 先完成无精修视觉的真实内容原型；
9. 完成视觉系统和雕像合同，灰模必须进站做固定镜头截图；
10. 只有 G0–G6 通过后才进入 L0–L2 集成；
11. 完成固定镜头、手机、昼夜、权利、构建和回归 QA；
12. 提交交付清单、回退证据、维护说明和复盘；
13. 更新工作流中经过样板验证的数值、模板和失败案例。

若输入不足，Agent 应输出明确的缺口与影响，不能用虚构内容补齐。若需求越过 L2，Agent 应登记变更并继续推进不受影响的 L0–L2 项，而不是自行扩张范围。

## 22. v0.1 待样板验证的假设

以下内容目前来自现有系统能力与方法设计，必须用首个真实人物样板验证后才能升级为正式标准：

1. **内容密度假设**：3 个旗舰项目是否通常足够；在第四个项目提供独立职业转向或能力证据时，3–4 个项目配合 4–6 个 Lab 和 5–7 个履历节点，是否仍不会产生信息负担。
2. **社交降级假设**：Social 区只保留通用模型与联系占位，是否不会明显削弱人物的可信度和完整性。
3. **雕像锚点假设**：在不增加新场景的情况下，一座人物雕像加 Landing 核心命题，是否能成为最强的人物识别锚点。
4. **固定镜头假设**：`+X / +Z` 观看象限、出生点、主行车线和手机竖/横屏五视点，是否足以覆盖雕像朝向风险。
5. **视觉差异假设**：仅用 L1 色彩、字体、UI/贴图和 L2 雕像/代表物，是否能让客户版本摆脱模板感，而不必进入 L3 新玩法。
6. **证据结构假设**：三图六幕项目模板是否适配不同职业；对于缺乏可视化产物的管理、咨询和服务工作，何种信息重建最有效。
7. **Lab 真实性假设**：客户是否愿意公开未完成问题、矛盾和失败，而不是只展示成就；访谈提问需要如何调整。
8. **评审成本假设**：命题、内容、灰模、视觉、发布五类签收是否恰好能阻止晚期返工，或仍然过多/过少。
9. **资产预算假设**：入口雕像的面数、材质、贴图和压缩体积应以当前站点哪一组实测基线为准，尚需样板构建和手机性能数据冻结。
10. **维护假设**：事实 ID、区域映射和资产权利清单能否显著降低下一位客户的重复劳动；应记录实际复用率和出错点。

样板完成后，复盘必须逐条给出“成立 / 需修订 / 被否定”，再发布 v0.2。

## 23. 本流程的唯一下一验收检查点

选定首个样板人物后，完成并签收以下五件套：

1. 候选评分与选择理由；
2. 达到最低条件的事实/授权账本；
3. 一句话人物核心命题；
4. Landing、Career、3 个 Projects、4–6 个 Lab 的人生—空间映射；
5. 一份严格限定 L0–L2 的视觉系统与人物雕像灰模合同。

这五件套通过前，不进入网站代码修改、最终模型精修或公开发布。
